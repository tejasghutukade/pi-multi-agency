Principles

Overview

  • You talk only to the Orchestrator.
  • The Orchestrator routes each request to the right specialist.
  • Sub-agents inherit behavior from compound-engineering skills.
  • Every sub-agent runs in its own cmux pane — one visible pi session per specialist.
  • Inter-agent messaging: hybrid file bus + cmux notify (pi-intercom demoted after live-run friction).
  • v0.3: pi lifecycle bridge frees the Orchestrator after delegate (no blocking agency_wait); push/queue specialist reports into hub chat.
  • Hub lock: Orchestrator must not implement product work (tools + persona bans).
  • Sub-agents may talk to each other under an explicit allowlist; all report back to the Orchestrator.
  • The Orchestrator may relay or mediate between agents when needed.
  • Agents can be temporary or persistent — Orchestrator decides per agent.

Execution

Runtime and infrastructure

Each specialist is a pi session in a cmux pane. The Orchestrator also runs in cmux. Agents do not share one terminal.

cmux

One pane per agent

Vertical tabs, notification panel, CLI/socket API for automation. Session restore + pi hooks (cmux hooks setup) let persistent agents resume with pi --session <id>.

Hybrid bus

Files + notify

Delegation/reports as JSON under .pi/agency/inbox/. cmux notify / pane flash for attention. Not a cmux message bus — cmux has no agent↔agent mailbox.

Policy

Spawn rules (v0)

Only the Orchestrator opens, reuses, promotes, or releases panes. Phase 2 control: agency_spawn / agency_ctl.py + sessions.json.

Open vs reuse

Prefer idle persistents

Idle healthy instance of role → reuse. Stale/dead pane rows cleared by reconcile. Cap at 6 specialist panes (spawn.maxSpecialistPanes).

Writers + twins

Work exclusive · Plan twin ok

Work never twins (allowWorkTwin: false) — queue. Busy Plan may spawn one temp plan-t* twin (allowPlanTempTwin: true, maxTempTwinsPerRole: 1).

Core design

Sub-agent core architecture

We will not use pi-subagents. Custom orchestration layer — inspired by task contracts and named sessions, without chains, fork context, worktrees, or built-in agent roster.

  • Persona — CE skill–grounded specialist role (Scout = custom skill)
  • Process — dedicated pi session in a cmux pane
  • Address — stable instance / inbox name for bus delegate / report

Process truth

pi agent_* events

agent_start → working · agent_settled → idle. Reliable per pane; extension hooks write sessions.json.

Task truth

Bus report / ask

Only a hybrid-bus report/ask for a taskId completes work to the Orchestrator. Settle ≠ done.

Silent settle

One nudge → abandon

Settled + no report → ~60s grace → one nudge. No agent_start after nudge → release + spawn + re-delegate same taskId.

Hub delivery

Push or queue

Hub idle → push full message into Orchestrator chat. Hub working → queue + banner; deliver on hub agent_settled.

Lifecycle

Sub-agent lifecycle

Spawn rules = how. Heuristics = when. Prefer reuse of idle persistents; Work/Plan default persistent; Scout/reviews default temporary.

Temporary

One-shot / burst

Scout, Brainstorm, Debug, reviewers by default. Tear down after task. Plan may get one temp twin when persistent Plan is busy.

Persistent

Multi-step + NOTES

Plan and Work by default. Keep pane; mark idle between tasks. Survive via pane context + NOTES.md + artifacts (see Memory).

  • Reuse idle persistent before spawning.
  • Never a second Work instance (allowWorkTwin: false).
  • Busy Plan: one plan-t* twin allowed under the pane cap.
  • Golden path: temp Scout → temp Brainstorm → persistent Plan (second Plan task proves reuse) — proven twice (Phase 1 + C+D).
  • Release persistents when the workflow ends (or user asks).

Attention
