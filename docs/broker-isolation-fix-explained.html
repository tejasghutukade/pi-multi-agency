MULTI-AGENCY · PR #4

What we fixed: brokers staying inside their own project

A plain-English guide to the project-isolation bug and the changes that close it. No prior knowledge of this repo required.

Read this first (30 seconds)

  • Multi-Agency lets one "hub" AI coordinate several "specialist" AIs. They talk through a local broker (a tiny message server on your machine).
  • The bug: the broker wasn't reliably tied to the project folder that owned it. Agents from a different checkout or worktree could join the wrong broker, or two brokers could collide.
  • The fix: every broker now belongs to exactly one project, every message is tagged with that project's ID, and every agent is told its project at startup. Cross-talk is no longer possible.

The mental model (the analogy)

Think of each project folder as an office. Inside an office, the team talks over an internal intercom (the broker). The hub is the team lead; specialists are teammates in other rooms.

Office A — project "alpha"
Its own intercom box.
Only alpha's team can speak on it.
Office B — project "beta"
Its own intercom box.
Only beta's team can speak on it.

The bug was like the two intercoms being cross-wired: someone in Office B could accidentally hear Office A's meetings, or both offices fought over one shared box. Our fix re-labels every intercom with its office name and makes sure each box is installed inside its own office.

What was actually broken

  • The code sometimes picked the wrong "home folder" for the broker — e.g. a nested sub-folder, a sibling worktree, or a reference repo you were just scanning.
  • Messages were not clearly stamped with the project they belonged to, so a "scout" running inside a different checkout could end up on the wrong broker.
  • Agents were expected to set their project environment after they had already started, which doesn't work — a running process can't repair its own environment from a chat command.
  • The broker's connection address (a "socket" file) could get too long in deep folders and simply fail to start.
  • The "only one broker at a time" lock could be stolen or leaked, letting two brokers fight over the same files.

Before vs after

Before (buggy)

  • Broker home decided loosely; can land outside the real project.
  • Messages not project-tagged → cross-project chatter possible.
  • Agent told its project mid-conversation (too late).
  • Socket path = full folder path → can break in deep folders.
  • Startup lock can expire a live owner or leak a process.

After (fixed)

  • Broker home = the true project root; wrong roots are rejected.
  • Every message stamped agency:<project>:<name> → strict separation.
  • Agent told its project before it starts (env vars at launch).
  • Socket in a short, private, owner-only folder → always starts.
  • Lock uses owner tokens; live owners protected; children reaped on failure.

What changed, in plain English

  1. Find the real project. We added one clear rule for "where does this project live?" — prefer the initialized project, resolve symlinks to their true location, fall back to the package root, and refuse if the agency folder escapes outside the project.
  2. Tag every conversation with the project. Broker session IDs now look like agency:<projectKey>:<agentName>. Two projects can never share a session, but replacing an agent with the same name within the same project still works.
  3. Tell agents their project at launch. The command that spawns each agent now sets AGENCY_ROOT and AGENCY_PROJECT_ROOT before the pi process starts. (Typing export in a chat later can't fix it — so we moved it earlier.)
  4. Short, safe connection address. Instead of a socket file buried in the deep project path, we use a small private folder like /tmp/pi-agency-<you>/<project>.sock with owner-only permissions. Project data still lives under .pi/agency.
  5. Safer broker startup. The startup lock now carries an owner token. A broker that is still alive is never declared "stale," only the true owner can release the lock, and if startup fails we clean up the spawned process instead of leaving it running.
  6. Hub connects right when you claim it. Running /agency-claim now immediately connects the broker and reports status, so the documented order is: /agency-claim → /agency-broker-status → agency_list.

Docs, charters, README, and the orchestrator skill were updated to match this new, correct sequence.

Key terms

TermWhat it means here
Project / project rootThe folder containing your code and its .pi/agency setup. The "office" in the analogy.
BrokerA tiny local server that delivers messages between the hub and specialists. The "intercom."
HubThe coordinator agent (usually named orchestrator) that delegates work.
SpecialistA worker agent spawned by the hub (Planner, Worker, Researcher, etc.).
Transport ID / session IDThe unique address a broker uses for one agent, now prefixed with the project key.
WorktreeA separate working copy of the same repo (common with git). Each must have its own broker.
SocketA file on disk that acts as the broker's "phone line" for local processes.
LockA marker file that ensures only one broker starts for a project at a time.

Show me the one-line essence

Every broker and every message is now scoped to the project that owns it:

agency:<projectKey>:<agentName>

If the project key differs, the agents simply cannot see each other — which is exactly the isolation we wanted.

Good to know (not bugs): This protects trusted, same-user local sessions from accidental collision. It is not a security wall against a hostile process on the same machine. Windows-specific named-pipe behavior still needs a manual check on a Windows machine; everything else is covered by automated tests (37 broker tests + 65 Python tests).
